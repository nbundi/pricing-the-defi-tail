## 1. Vulnerability Overview

Curve Finance operates StableSwap and Cryptoswap AMM pools written in the Vyper language. On July 30, 2023, a compiler bug was discovered in Vyper compiler versions 0.2.15, 0.2.16, and 0.3.0 where the `@nonreentrant` decorator failed to correctly generate reentrancy lock code, resulting in simultaneous attacks on multiple Curve pools.

The core vulnerability is a combination of two factors:

1. **Vyper Compiler Bug**: The `@nonreentrant("lock")` decorator failed to correctly insert reentrancy guard code into EVM bytecode in certain compiler versions. As a result, the lock variable was never actually checked or set at runtime, allowing reentrancy.

2. **ETH Transfer Callback Exploitation**: The `add_liquidity` and `remove_liquidity` functions transfer ETH to `msg.sender` during native ETH deposits/withdrawals. This transfer triggers the attacker's `receive()` function, enabling a re-call of `add_liquidity` through an abnormal reentrancy path, allowing LP tokens to be minted twice.

Affected major pools:
- **pETH/ETH** (0x9848...5C5): ~$11.5M loss
- **alETH/ETH** (0xC4C3...): ~$13.6M loss
- **msETH/ETH**: Damage minimized by whitehat rescue
- **CRV/ETH** (0x8301...511): Additional attack attempts

## 2. Vulnerable Code Analysis

### 2.1 Vyper @nonreentrant Decorator — Compiler Bug (Core Vulnerability)

**Vulnerable Vyper code (compiled with Vyper 0.2.15 ~ 0.3.0)**:

```vyper
# ❌ Vulnerable: this decorator fails to generate correct reentrancy guard bytecode
@nonreentrant("lock")
@external
@payable
def add_liquidity(amounts: uint256[N_COINS], min_mint_amount: uint256) -> uint256:
    # Internal logic ...
    # ETH refund transfers to msg.sender → triggers attacker's receive()
    if use_eth:
        raw_call(msg.sender, b"", value=eth_value)  # ❌ Reentrancy possible at this point
    # LP token minting completes after the call
    return mint_amount
```

**EVM bytecode that a correct compiler should generate (pseudocode)**:

```solidity
// ✅ Correct reentrancy guard — code that a proper compiler should insert
// Store lock state in a storage slot
uint256 constant LOCK_SLOT = keccak256("vyper.reentrancy_lock");

modifier nonReentrant() {
    // ✅ On entry: verify lock slot is empty, then set it
    require(storageLoad(LOCK_SLOT) == 0, "Reentrancy blocked");
    storageStore(LOCK_SLOT, 1);
    _;
    // ✅ On exit: release lock
    storageStore(LOCK_SLOT, 0);
}
```

**EVM bytecode actually generated by the buggy version (pseudocode)**:

```solidity
// ❌ Compiler bug: lock slot check/set code is missing or misplaced
// As a result, the lock state always reads as 0 (unlocked), allowing reentrancy
modifier nonReentrant_BROKEN() {
    // ❌ No lock check code, or incorrect slot reference
    _;
    // ❌ No lock set code
}
```

**Problem**: A bug exists in the internal code generation logic of Vyper compiler versions 0.2.15, 0.2.16, and 0.3.0 during `@nonreentrant` processing — specifically in the lock variable's slot calculation or bytecode insertion order — completely disabling the reentrancy guard at runtime. This is a supply-chain level vulnerability where security appears to be applied at the source code level, but the actually deployed bytecode contains no defensive logic.

### 2.2 add_liquidity + remove_liquidity Reentrancy Path

**Vulnerable Vyper code (CurvePool — based on pETH/ETH pool, reconstructed)**:

```vyper
# ❌ Vulnerable: @nonreentrant decorator does not actually function
@nonreentrant("lock")
@external
@payable
def add_liquidity(amounts: uint256[2], min_mint_amount: uint256) -> uint256:
    # Liquidity ratio calculation
    d1: uint256 = self._A_gamma()
    # ...
    # Balance updates precede the process of depositing ETH into the pool
    # ...
    # ❌ ETH refund can occur before LP token minting
    # → attacker's receive() executes at this point, enabling reentrancy
    mint_amount: uint256 = self._calc_token_amount(amounts, True)
    CurveToken(self.token).mint(msg.sender, mint_amount)  # LP token mint
    return mint_amount

@external
def remove_liquidity(token_amount: uint256, min_amounts: uint256[2]):
    # msg.sender.call{value}() occurs when returning ETH
    # → attacker's receive() can reenter with already-minted LP tokens
    for i in range(N_COINS):
        if i == 0:
            raw_call(msg.sender, b"", value=amounts[0])  # ❌ ETH transfer → reentrancy trigger
        # ...
```

```vyper

@external
@payable
def add_liquidity(amounts: uint256[2], min_mint_amount: uint256) -> uint256:

    # ✅ Lock actually works, so reentrancy immediately reverts
    # ...
    mint_amount: uint256 = self._calc_token_amount(amounts, True)
    CurveToken(self.token).mint(msg.sender, mint_amount)
```

## 3. Attack Flow

### 3.1 Preparation Phase

- The attacker deploys a dedicated attack contract (pETH pool attack: `0x466b...f04`, CRV/ETH pool attack: `0x83e0...0c1`)
- The attack contract implements a `receive()` function that triggers reentrancy
- A `nonce` variable tracks the number of reentries, executing reentrancy only at specific points

### 3.2 Execution Phase — pETH/ETH Pool Attack (exp01)

1. **Flash loan acquisition**: Flash loan 80,000 WETH from Balancer Vault
2. **WETH → ETH conversion**: Call `WETH.withdraw()` to obtain 80,000 ETH
3. **1st add_liquidity**: Deposit 40,000 ETH into CurvePool to obtain LP tokens
4. **1st remove_liquidity call (reentrancy entry point)**:
   - CurvePool transfers ETH to attacker → triggers `receive()`
   - Branch on `nonce == 0` condition inside `receive()`
5. **During reentrancy — 2nd add_liquidity**: Deposit an additional 40,000 ETH inside `receive()` → obtain additional LP tokens (permitted due to lock malfunction)
6. **1st remove_liquidity completes**: Return from reentrancy and complete remove_liquidity → LP tokens burned, ETH received
7. **Nonce increment**: Verify double-minted LP token balance
8. **2nd remove_liquidity**: Drain additional ETH with the surplus double-minted LP tokens (equivalent to 10,272 ETH)
9. **pETH → ETH swap**: Exchange obtained pETH for ETH
10. **ETH → WETH conversion**: Convert profit
11. **Flash loan repayment**: Return 80,000 WETH to Balancer
12. **Profit realization**: ~$11.5M USD worth of ETH obtained

### 3.3 Execution Phase — CRV/ETH Pool Attack (exp02)

20-iteration loop (repeated attack in smaller units):
1. Flash loan 10,000 WETH from Balancer
2. WETH → ETH conversion
3. Loop (20 iterations):
   - `add_liquidity(400 ETH)` → `remove_liquidity` (reentrancy trigger) → execute `add_liquidity` + `exchange` reentrancy
   - `remove_liquidity_one_coin` → receive ETH
   - `exchange(CRV→ETH)` → receive additional ETH
4. ETH → WETH conversion, flash loan repayment

## 5. Vulnerability Classification

| ID | Vulnerability | Severity | CWE | Matching Pattern | Similar Cases |

| V-02 | ETH transfer callback reentrancy | CRITICAL | CWE-841 | 01_reentrancy.md | Fei/Rari (2022) |
| V-03 | LP token double-minting | CRITICAL | CWE-362 | 16_accounting_sync.md | Cream Finance (2021) |

### V-01: Vyper Compiler Bug — @nonreentrant Disabled

- **Description**: A bug exists in the code generation logic for processing the `@nonreentrant` decorator in Vyper compiler versions 0.2.15, 0.2.16, and 0.3.0, causing the reentrancy lock to not actually function at runtime. While security appears to be applied at the source code level, the actual deployed bytecode lacks the defensive logic.
- **Impact**: All Curve pools that relied on the compiler and applied `@nonreentrant` (and other protocols deployed with the same compiler versions) were exposed to reentrancy attacks.
- **Attack conditions**: Contracts compiled with Vyper 0.2.15~0.3.0, external calls involving ETH transfers, attacker calling through a contract.

### V-02: ETH Transfer Callback Reentrancy

- **Description**: The `add_liquidity` and `remove_liquidity` functions of Curve pools handle native ETH directly. When returning ETH, they use `raw_call(msg.sender, ...)` or `msg.sender.call{value: ...}()`, which executes the attacker's `receive()` function. Combined with V-01, `add_liquidity` can be re-called during `remove_liquidity` execution, resulting in double-minting of LP tokens.
- **Impact**: Excessive LP tokens minted relative to pool assets enable draining the entire pool.
- **Attack conditions**: Functions handling native ETH, `receive()` callback implementation, reentrancy lock not functioning.

### V-03: LP Token Double-Minting

- **Description**: Due to reentrancy, `add_liquidity` is re-called before `remove_liquidity` completes, causing LP tokens to be minted twice with the same ETH. Since the first minted LP has not yet been burned when the second LP is created, the attacker holds more LP tokens than the ETH actually deposited.
- **Impact**: Mismatch between pool reserves and LP token supply causes pool insolvency.
- **Attack conditions**: Automatically occurs when V-01 + V-02 are combined.

### V-04: Supply Chain Vulnerability (Compiler)

- **Description**: The fundamental uniqueness of this vulnerability is that the bug existed in the compiler toolchain rather than in the smart contract source code itself. Even though developers used correct security practices (`@nonreentrant`), the compiler generated incorrect bytecode, creating the vulnerability. Traditional audits would struggle to detect this at the source code level.
- **Impact**: Potential impact on all protocols sharing the affected compiler version. Exposes the limitations of auditing.
- **Attack conditions**: Use of specific Vyper versions.

### Immediate Actions


```bash

# ✅ Specify compiler version in source

```

#### Action 3: Enforce CEI pattern (Vyper)

```vyper
# ✅ Fixed: finalize LP token mint/burn state before ETH transfer
@nonreentrant("lock")  # ✅ Functions correctly with proper compiler
@external
@payable
def add_liquidity(amounts: uint256[2], min_mint_amount: uint256) -> uint256:
    # ✅ 1) Checks: validate inputs
    assert amounts[0] > 0 or amounts[1] > 0, "Zero amounts not allowed"

    # ✅ 2) Effects: finalize state changes first
    mint_amount: uint256 = self._calc_token_amount(amounts, True)
    CurveToken(self.token).mint(msg.sender, mint_amount)  # ✅ LP minted before ETH transfer

    # ✅ 3) Interactions: external calls last
    if use_eth and refund_amount > 0:
        raw_call(msg.sender, b"", value=refund_amount)  # ✅ ETH refund after LP minting

    return mint_amount
```

#### Apply Pull Pattern (instead of direct ETH transfer)

```vyper
# ✅ Pull pattern: change from direct ETH transfer to claim-based approach
pending_eth: HashMap[address, uint256]

@nonreentrant("lock")
@external
def remove_liquidity(token_amount: uint256, min_amounts: uint256[2]):
    # After state changes are finalized, record ETH rather than transferring directly
    # ...
    self.pending_eth[msg.sender] += eth_amount  # ✅ Record only (no external call)

@nonreentrant("lock")
@external
def claim_eth():
    amount: uint256 = self.pending_eth[msg.sender]
    self.pending_eth[msg.sender] = 0  # ✅ Clear state first
    raw_call(msg.sender, b"", value=amount)  # ✅ Transfer after
```

## 7. Lessons Learned

1. **Do not trust the compiler (Zero Trust Toolchain)**: Even when security practices are correctly implemented in source code, compiler bugs can cause the deployed bytecode to differ. Critical contracts should have their deployed bytecode directly verified, or formal verification tools should be used.

2. **Recognize the limits of auditing**: Traditional source code audits cannot detect compiler-level bugs. Security audits at the deployed bytecode level and EVM trace analysis are additionally required.

3. **CEI pattern as a secondary defense against reentrancy**: Even when `nonReentrant` fails to function, correctly implementing the CEI (Checks-Effects-Interactions) pattern can minimize the impact of reentrancy attacks. A defense-in-depth strategy applying both is necessary.

4. **Native ETH handling increases risk**: Handling native ETH directly instead of WETH causes the `receive()` callback to execute, increasing the reentrancy attack surface. Where possible, use WETH or a pull pattern to minimize external calls.

5. **Supply Chain Security**: Pin the versions of compilers, libraries, and dependencies in use and periodically monitor security vulnerability announcements. Subscribe to security advisory channels for language teams such as Vyper and Solidity.

6. **The reality of whitehat competition**: Immediately after vulnerability disclosure, multiple attackers simultaneously attempted attacks, and some pools had funds rescued by whitehats who acted first. A protocol pause mechanism that activates immediately upon vulnerability discovery is essential to minimize damage.

7. **Invariant monitoring**: Real-time tracking of invariants of the form `totalSupply(LP) == f(poolReserves)` via on-chain monitoring systems (e.g., Forta) can detect anomalies early in an attack.

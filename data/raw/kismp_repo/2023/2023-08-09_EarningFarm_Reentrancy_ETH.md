## 1. Vulnerability Overview

Earning Farm is a leveraged yield farming protocol built on top of Aave. Users deposit ETH into the **EFVault** contract to receive **ENF_ETHLEV** share tokens; redeeming those shares returns ETH along with yield generated by the leverage strategy.

The core vulnerability lies in the `withdraw()` function's structure: **ETH is transferred to the receiver first via an external contract (Controller)**, and shares are burned only afterward based on the result. This flow violates the **CEI (Checks-Effects-Interactions) pattern**.

During the ETH transfer, the attacker's `receive()` function executes at a point when the share balance has not yet been decremented, enabling the attacker to:
1. Transfer shares to another contract, and
2. Complete the original withdrawal — **double-collecting ETH against the same shares**

The attacker repeatedly leveraged Uniswap V3 flash swaps (10,000 ETH each) until the protocol's `totalAssets` dropped below 1 ETH.

## 2. Vulnerable Code Analysis

### 2.1 `EFVault.withdraw()` — CEI Pattern Violation (Core Vulnerability)

**Vulnerable Code (❌)**:
```solidity
// ❌ Vulnerability: share burn occurs AFTER external call (withdraw) — CEI pattern violation
function withdraw(
    uint256 assets,
    address receiver
) public virtual nonReentrant unPaused returns (uint256 shares) {
    // [1] Calculate share ratio held
    shares = (totalSupply() * assets) / totalAssets();

    // [2] ❌ ETH is transferred to receiver first via external contract (Controller)
    //     receiver's receive() function executes during this call
    (uint256 withdrawn, uint256 fee) = IController(controller).withdraw(assets, receiver);

    // [3] ❌ Share burn happens only after ETH transfer — reentrancy has already occurred
    if (balanceOf(msg.sender) < shares) {
        shares = balanceOf(msg.sender);  // balance may already be near 0
    }
    _burn(msg.sender, shares);  // burn is too late
}
```

**Issues**:
- Despite the `nonReentrant` modifier, `IController(controller).withdraw()` transfers ETH directly to `receiver`, triggering receiver's `receive()` callback.
- `nonReentrant` only protects against **reentrancy into the same contract**; if the ETH recipient transfers shares to a **separate contract (Exploiter)** and that contract calls `withdraw()` again, the lock is bypassed.
- Because `_burn` occurs after the ETH transfer, if shares are moved to another address in the prior step, `balanceOf(msg.sender)` is already exhausted and only the residual amount is burned rather than the correct full share quantity.

**Fixed Code (✅)**:
```solidity
// ✅ Fix: Apply CEI pattern — state change (share burn) performed before external call
function withdraw(
    uint256 assets,
    address receiver
) public virtual nonReentrant unPaused returns (uint256 shares) {
    // [1] Calculate share ratio held
    shares = (totalSupply() * assets) / totalAssets();

    // [2] ✅ Effect first (before external call): burn shares
    if (balanceOf(msg.sender) < shares) {
        shares = balanceOf(msg.sender);
    }
    _burn(msg.sender, shares);  // burned first — no balance left if reentered

    // [3] ✅ Interaction after state change: transfer ETH
    (uint256 withdrawn, uint256 fee) = IController(controller).withdraw(assets, receiver);
}
```

### 2.2 `nonReentrant` Bypass — Cross-Contract Reentrancy

**Vulnerable Flow (❌)**:
```solidity
// ❌ Vulnerability: nonReentrant only blocks reentrancy within the same contract
// If the attacker transfers shares to an external contract (Exploiter) in receive()
// and Exploiter.withdraw() calls ENF_ETHLEV.withdraw() again, the lock is bypassed

// [Attacker contract's receive()]
receive() external payable {
    if (msg.sender == Controller) {
        // ❌ On ETH receipt — shares transferred to Exploiter contract
        ENF_ETHLEV.transfer(
            address(exploiter),
            ENF_ETHLEV.balanceOf(address(this)) - 1  // transfer all but 1 share
        );
        nonce++;
    }
}
```

**Fixed Code (✅)**:
```solidity
// ✅ Fix: Lock share transfers during withdrawal (pool-based separation)
// Or apply an additional lock that blocks Transfer events while withdraw() is executing

// Method 1: withdrawing flag
bool private _withdrawing;

modifier noTransferDuringWithdraw() {
    require(!_withdrawing, "transfer locked during withdraw");
    _;
}

function withdraw(...) external nonReentrant {
    _withdrawing = true;
    // ... logic ...
    _withdrawing = false;
}

// ERC20 _transfer override
function _transfer(...) internal override noTransferDuringWithdraw {
    super._transfer(...);
}
```

## 5. Vulnerability Classification

| ID | Vulnerability | Severity | CWE |
|----|--------|--------|-----|
| V-01 | CEI pattern violation — state change after external call | CRITICAL | CWE-841 |
| V-02 | Cross-contract reentrancy (`nonReentrant` bypass) | CRITICAL | CWE-1265 |
| V-03 | Capital amplification via flash loan | HIGH | CWE-400 |
| V-04 | Share burn validation logic flaw (post-hoc `balanceOf` check) | HIGH | CWE-362 |

### V-01: CEI Pattern Violation

- **Description**: The `withdraw()` function transfers ETH through an external contract (Controller) first, then burns shares based on the transfer result — an Interactions-before-Effects inversion
- **Impact**: During the ETH transfer, shares can be exfiltrated via the receiver's callback, reducing the burn quantity to 1 and allowing double collection of ETH against the same shares
- **Attack Condition**: Attacker can supply a callback-capable contract address as the `receiver` of `withdraw()`

### V-02: Cross-Contract Reentrancy (`nonReentrant` Bypass)

- **Description**: The `nonReentrant` modifier only blocks direct reentrancy into the same contract. If the ETH recipient calls `ENF_ETHLEV.withdraw()` via a **separate Exploiter contract**, it passes through without any lock.
- **Impact**: Two contracts (ContractTest + Exploiter) can each execute withdrawals against the same set of shares
- **Attack Condition**: Attacker controls two or more contracts and can transfer shares in real time

### V-03: Capital Amplification via Flash Loan

- **Description**: 10,000 ETH borrowed via Uniswap V3 flash swap maximizes attack impact against the protocol's asset pool with minimal actual capital
- **Impact**: Attacker can drain the protocol's entire assets without holding real capital
- **Attack Condition**: A liquidity pool with flash loan capability exists; attack completes within a single transaction

### V-04: Share Burn Validation Logic Flaw

- **Description**: The `if (balanceOf(msg.sender) < shares)` check in `withdraw()` occurs immediately before the burn; if shares have already been transferred out, only the remaining balance (1) is burned
- **Impact**: Attacker receives ETH for a large share position, immediately transfers shares to evade the burn
- **Attack Condition**: Automatically satisfied when combined with V-01 and V-02

## 6. Remediation Recommendations

### Immediate Actions

**[1] Apply CEI Pattern — Move State Changes Before External Calls**

```solidity
// ✅ Fixed withdraw() — CEI pattern compliant
function withdraw(
    uint256 assets,
    address receiver
) public virtual nonReentrant unPaused returns (uint256 shares) {
    // [C] Checks: input validation
    require(assets > 0, "zero assets");

    // [E] Effects: state changes first — burn shares
    shares = (totalSupply() * assets) / totalAssets();
    if (balanceOf(msg.sender) < shares) {
        shares = balanceOf(msg.sender);
    }
    _burn(msg.sender, shares);  // ← burned before external call

    // [I] Interactions: external call — transfer ETH
    (uint256 withdrawn, uint256 fee) = IController(controller).withdraw(assets, receiver);
}
```

**[2] Block Share Transfers During Withdrawal — Transfer Lock**

```solidity
// ✅ Block ERC20 Transfer while withdraw is executing
bool private _inWithdraw;

function withdraw(...) public nonReentrant {
    _inWithdraw = true;
    // ... logic ...
    _inWithdraw = false;
}

function _beforeTokenTransfer(address from, address to, uint256 amount)
    internal override
{
    // Block share transfers from the withdrawing address
    require(!_inWithdraw || from == address(0), "transfer locked during withdraw");
    super._beforeTokenTransfer(from, to, amount);
}
```

**[3] Switch ETH Transfer to Pull Pattern**

```solidity
// ✅ Pull instead of push — eliminates attacker callback opportunity
mapping(address => uint256) public pendingWithdrawals;

function withdraw(uint256 assets, address receiver) public nonReentrant {
    // Burn shares
    _burn(msg.sender, shares);

    // Register in withdrawal queue instead of transferring ETH immediately
    pendingWithdrawals[receiver] += assets;
}

// Separate function to claim ETH
function claimWithdrawal() external nonReentrant {
    uint256 amount = pendingWithdrawals[msg.sender];
    require(amount > 0, "nothing to claim");
    pendingWithdrawals[msg.sender] = 0;
    (bool ok, ) = msg.sender.call{value: amount}("");
    require(ok, "ETH transfer failed");
}
```

### Structural Improvements

| Vulnerability | Recommended Action |
|--------|-----------|
| CEI pattern violation (V-01) | Move share burn before Controller call |
| Cross-contract reentrancy (V-02) | Block share Transfer events during withdrawal, or adopt pull pattern |
| Flash loan amplification (V-03) | Apply timelock or rate limiting for large deposits/withdrawals within a single block |
| Share burn validation flaw (V-04) | Fix burn quantity at withdrawal initiation based on input parameters; disallow recalculation |
| Overall architecture | Decouple Controller → Vault direct transfer structure into a withdrawal queue within the Vault |

## 7. Lessons Learned

1. **`nonReentrant` is not a complete defense**: OpenZeppelin's `ReentrancyGuard` only blocks reentrancy within the same contract. When a system is distributed across multiple contracts, cross-contract reentrancy — where an ETH transfer recipient calls another contract within the system — is not prevented.

2. **The CEI pattern is a non-negotiable baseline**: Checks-Effects-Interactions is the most fundamental defense against reentrancy attacks. All state changes must be completed before any external contract call (including ETH transfers).

3. **ERC20 token transfers must be blocked during the withdrawal process**: If share tokens can be transferred to a third party mid-withdrawal, the burn quantity calculation becomes meaningless. Either lock token transfers during withdrawal, or fix the burn quantity at the start of withdrawal and make it immutable.

4. **Pull pattern is safer than push pattern**: Transferring ETH immediately within a function call (push) creates an opportunity for the receiver's callback to execute. Designing a system where amounts are recorded in a withdrawal queue and users claim separately (pull) eliminates the attack surface entirely.

5. **Flash loans remove capital constraints from attackers**: When reentrancy is possible and flash loans are available, an attacker can effectively drain an entire protocol's assets with unlimited capital. Rate limits or timelocks on large deposits/withdrawals within a single block must be implemented.

6. **State changes after external calls must be flagged as a priority during code review**: During audits or code reviews, automated tools or checklists should flag any pattern where state variables are modified after external contract calls (`call`, `transfer`, `send`, interface calls).
